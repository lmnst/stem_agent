# stem-agent: write-up

> The deterministic primitive-search path produces every number in
> this document. There is no LLM in the system. Section 6 is explicit
> about that as a design choice and what a future LLM extension would
> look like; the agent does not carry an unmeasured parallel path.

## 1. Problem framing

I chose **single-function Python bug-repair** as the problem class.
Each task is a folder with `solution.py` (one buggy function, ~5 to 25
lines) and `test_solution.py` (pytest-compatible asserts). A task is
**solved** when every `test_*` callable in `test_solution.py` returns
without raising, run inside an isolated temp workspace.

Why this class fits the brief:

- **Narrow enough that "specialized" is concrete.** A specialized
  agent here means *which AST mutations to try first, with what
  budget, when to stop, and which workflow steps to run.* Those
  choices are measurable.
- **Automatically verified.** No graders, no LLM-judging; a
  subprocess runs the test file and exits 0 or non-zero.
- **Has recurring families.** Off-by-one, wrong operator, wrong
  constant, swapped args, wrong boolean op, etc.; eight in total in
  this benchmark. A generic agent has no reason to prefer one over
  another; a specialized agent can.

The benchmark, `benchmarks/pybugs/`, has 32 hand-authored tasks split
12 train / 8 dev / 12 test. The split is fixed in the directory tree
and is the same split used to generate the headline numbers below.

## 2. What the stem knows vs. what it discovers

The stem blueprint at `artifacts/stem_blueprint.json` is a thin
domain-specific scaffold: it is configured to attempt eight AST-level
mutation primitives in `sorted(PRIMITIVE_NAMES)` order, with budget 8.
The stem is *not* an undifferentiated agent. It already knows the
problem class is Python bug-repair and ships with the eight primitives
needed to solve in-bank tasks. What the stem does *not* know is
**which** primitive to prefer for which task, how generous a budget
the domain warrants, or how variant fan-out interacts with priority.
Those are the things evolution should discover.

The evolved blueprint at `artifacts/evolved_blueprint.json` differs
on three agent-read fields:

| field | stem | evolved | source of change |
|---|---|---|---|
| `primitive_priority` | alphabetical | reverse-alphabetical | dev-set selection |
| `primitive_budget` | 8 | 12 | profile-recommended budget (from train iters) |
| `early_stop_no_progress` | 8 | 12 | tied to budget |

The reverse-alphabetical winner is *not* a hand-chosen result; it is
the candidate that minimized mean iterations to solve on the dev split
(see `artifacts/evolution_log.json`). It happens to win because the
primitives at the head of the reverse list (`swap_true_false`,
`swap_eq_neq`) generate **few** variants on most sources, so they
"fall through" cheaply, and the next primitive on the list,
`swap_compare_strict`, often catches the bug in 1 to 2 attempts. The
priorities ranked by training fix-frequency (`shift_const_pm1`,
`swap_arith_pair`) generate *many* variants; useful when they're the
right primitive, costly when they're not.

Lesson: **primitive popularity != primitive search efficiency** when
each primitive's variant fan-out varies by an order of magnitude on
the same source.

## 3. Headline: budget-controlled comparison

The 4-row table below is the headline. It is generated by
`python -m stem_agent.cli compare ...` from the same artifacts
described above. Wilson 95% intervals from `stem_agent.stats` are on
every pass rate. *iters/all* averages over all 12 tasks, scoring
unsolved tasks at the effective per-task budget, so a blueprint cannot
flatter its mean by running fewer hard solves; *iters/solved* is the
classical metric. Both denominators are reported.

| row                    | budget | pass         | CI95            | iters/solved | iters/all |
|---|---|---|---|---|---|
| stem (default)         |   8    | 9/12 = 75.0% | [46.8, 91.1]    |        2.56  |     3.92  |
| stem (evolved budget)  |  12    | 11/12 = 91.7%| [64.6, 98.5]    |        3.91  |     4.58  |
| evolved (stem budget)  |   8    | 12/12 = 100% | [75.8, 100.0]   |        3.08  |     3.08  |
| evolved (default)      |  12    | 12/12 = 100% | [75.8, 100.0]   |        3.08  |     3.08  |

**What this says.** Most of the headline gap is *priority*, not
*budget*. The evolved priority finds every test bug inside 8
iterations (the stem's own budget); raising the stem's budget to 12
recovers two of three failures but still misses task_024 within
budget. The "75 -> 100" gap therefore survives budget control as
"75 -> 100 at the same budget," not as "75 -> 100 because the evolved
ran longer." With n=12 the Wilson intervals overlap, so this is a
suggestive single-run result, not a statistically-distinguishable
finding.

What the controlled comparison *also* says: the priority-and-budget
axis alone is not a deep specialization. A real specialization
mechanism is added in commit 2.

## 4. How domain analysis, candidate generation, and selection actually work

### Domain analysis (`stem_agent/analysis.py:analyze_domain`)

Two signals contribute to a `DomainProfile` (`artifacts/profile.json`):

1. **Empirical primitive frequency.** Run a uniform-priority probe
   over the train split with a generous budget. For every task the
   probe solves, the *primitive that produced the passing variant*
   is counted. Normalized counts become `primitive_frequencies`.
   This is observed evidence, not a hand-picked prior; the analyzer
   does not know which primitive *should* fix any given task.
2. **Iters-to-solve distribution.** The probe records iterations for
   each train solve. `recommended_budget` =
   `max(8, round(max_iters_observed * 1.5) + 4)`, the
   domain-aware budget the evolved blueprint uses.

What's hardcoded? **The set of primitives** (eight AST-level mutations
in `primitives.py`) and the **shape** of the blueprint. Nothing about
the specific tasks or their fixes is hardcoded.

### Candidate generation (`stem_agent/evolve.py:_initial_candidates`)

A small, explicit grid:

- Three priority orderings:
  - `ranked` by `profile.primitive_frequencies`
  - `alpha`: `sorted(PRIMITIVE_NAMES)` (no domain knowledge)
  - `reverse`: `sorted(PRIMITIVE_NAMES, reverse=True)` (adversarial control)
- Three budgets: `8`, `recommended_budget`, `recommended_budget * 2`.

Nine initial candidates. They differ on axes that demonstrably move
dev pass-rate or mean-iterations-to-solve.

### Selection and stopping (`stem_agent/evolve.py:evolve`)

Each generation, every candidate is scored on the dev split. The sort
key is
`(-pass_rate, mean_iters_solved, abs(budget - recommended_budget), budget)`:
pass rate first; ties broken by efficiency, then by closeness to the
profile-recommended budget (so a saturated dev pass-rate cannot reward
wasteful budget growth), then by smaller raw budget for determinism.
When a candidate has zero solves, `mean_iters` is set to infinity so
universally-failing candidates never beat anything on the iters
tiebreaker.

The next generation perturbs the top 2 survivors *expansively*: double
the budget (capped at `recommended_budget * 4` so a saturated dev
pass-rate cannot drive the persisted budget arbitrarily high), and
promote the profile's top-5 primitives to the head. Stopping rules:
stop when no strict improvement for 2 generations, or after a hard
cap of 3 generations.

## 5. What surprised me, what failed, what's next

**Surprise 1: "ranked-by-popularity" priority underperforms reverse-alphabetical on dev.**
I expected `ranked` (priority order = train-fix-frequency) to be the
winner. It wasn't. `shift_const_pm1` and `swap_arith_pair` are the
most-frequent fixers in train, but they also generate the most
variants on a typical source: putting them first means a buggy
file with 4 integer constants and 2 BinOps burns 12+ iterations
before the next primitive even runs. The lesson: **search-efficient
ordering is about variant fan-out, not just fix probability.**

**Surprise 2: a tighter budget on the stem still misses a test bug.**
Even at `b=12` (the evolved blueprint's own budget), the alphabetical
stem leaves task_024 (`signs_match`) unsolved within budget. The fix
needs `swap_and_or` on a compound boolean expression with several
preceding sites that the alphabetical priority enumerates first. So
priority is doing real work, not just budget.

**What was removed from the in-between state.**
The previous version carried two pieces of code that were live in the
agent loop but excluded from the headline numbers: an LLM proposal
path and a localization heuristic. Both have been removed. The LLM
extension point is documented in section 6 below; localization is a
deletion, not a deferral.

**With another week:** (a) replace the deterministic primitive bank
with a constraint-search that also handles multi-edit bugs (right now
every benchmark task is single-edit by construction); (b) bring an
LLM into the proposal step, on a measured row of the comparison
table, with cost reported alongside; (c) stress-test on tasks the
primitive bank cannot fix to verify the agent gives up cleanly; (d)
add a per-task primitive timing prior so cheap-on-this-source
information is not rediscovered each run.

## 6. Limitations and decisions

- **No LLM.** The agent does not call any external model. The previous
  draft of the system carried an `LLMClient` that could inject a
  proposal at index 0 of the variant queue, but its contribution to
  the headline was zero (the demo ran without an API key). To avoid
  the failure mode of an unmeasured second path, the LLM machinery
  has been removed. A future extension would slot in at the `propose`
  workflow step in `agent.py`: the natural insertion point is a new
  variant constructed from a model's reply, prepended to the variant
  list. That extension is described, not stubbed.
- **The benchmark is small (32 tasks) and authored.** The bug
  families are realistic, but I did not draw from a public bug
  corpus. Results here should not be read as claims about general
  program-repair performance.
- **The primitive set is closed.** Eight AST mutations cover the bug
  families I encoded but do not handle, e.g., wrong-loop-bound bugs
  on identifier-typed bounds, missing edge-case branches, or anything
  multi-line. The agent has no escape valve; out-of-bank tasks fail
  cleanly.
- **Evolution searches a small grid, not a large space.** With nine
  initial candidates per run, this is closer to *grid selection with
  perturbations* than to evolutionary search in the ML sense. I think
  that's the right complexity level for the candidate axes that
  exist; commit 2 introduces a separate learned mechanism that is not
  on this grid.
- **The win on test depends on a deliberately conservative stem.** A
  hand-tuned stem could solve more than 9/12 with a larger budget.
  The point of the comparison is not "evolution beats hand tuning";
  it's "evolution beats *the agent before it has seen any data*."

## 7. Where to look in the code

- Initial state: `stem_agent/evolve.py:stem_blueprint`
- Domain analysis: `stem_agent/analysis.py:analyze_domain`
- Candidate generation: `stem_agent/evolve.py:_initial_candidates`,
  `_next_generation`
- Selection and stopping: `stem_agent/evolve.py:evolve`,
  `CandidateScore.key`
- Agent loop: `stem_agent/agent.py:solve_task`
- Workflow vocabulary and validation: `stem_agent/blueprint.py:WORKFLOW_STEPS`,
  `validate_workflow`
- Sandboxed runner: `stem_agent/runner.py`
- Mutation primitives: `stem_agent/primitives.py`
- Wilson 95% CI helper: `stem_agent/stats.py`
- Budget-controlled compare: `stem_agent/cli.py:cmd_compare`

The pipeline writes the following under `artifacts/` (regenerated by
the documented commands; see `README.md`):

- `profile.json`: the `DomainProfile` from train probing.
- `stem_blueprint.json`: the domain-agnostic baseline blueprint.
- `evolved_blueprint.json`: the dev-set winner, persisted under name
  `evolved` with a `lineage` field tracing the candidate it came from.
- `evolution_log.json`: per-generation, per-candidate scoring trace
  with the full blueprint of every candidate.
- `stem_test.json`, `evolved_test.json`: per-task evaluation of each
  blueprint on the held-out test split.
- `compare_test.json`: the four-row budget-controlled table from
  section 3, with per-blueprint per-task records.
